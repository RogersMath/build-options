Platform by platform

The reasoning behind each band, with sources. The left tile in every dimension is the default; the right is what tuning buys you.

01

WordPress

Dynamic CMS · PHP + database

The default of the web — about 43.5% of all sites run on it, which is precisely why it is the default target. [3]

SEO

DefaultModerate

Stock themes ship render-blocking CSS/JS and per-request database calls; out-of-the-box Lighthouse often lands below 90, dragging the Core Web Vitals that feed Google's page-experience signal. [4, 6]

TunedStrong

Caching, a CDN, image optimization and a lean theme clear the CWV thresholds. The CMS is not the ceiling — the default configuration is. [4, 5]

AEO

DefaultWeak

Theme markup is deep and div-heavy, burying the self-contained answer that LLMs try to extract in chunks. [11, 12]

TunedStrong

Schema plugins (FAQPage, Article) plus answer-first copy give strong machine-readability — at the price of more plugins, i.e. more surface. [12, 14]

Cost

DefaultModerate

Bare-bones shared hosting runs roughly $75–150/yr, but 'cheap' buys speed and security debt you repay later. [17]

TunedWeak

A properly run site — managed hosting, premium plugins, security — carries $600–$12,000/yr in maintenance TCO. Here optimization raises cost. [15, 16]

Security

DefaultPoor

11,334 ecosystem vulnerabilities in 2025 (91% in plugins), ~13,000 sites hacked daily, and exploits landing within ~5 hours of disclosure. A stock install is an automated target. [1, 3]

TunedModerate

Hardening, a WAF, minimal plugins and auto-updates help — but ~57% of flaws need no authentication and traditional WAFs miss most WordPress-specific attacks. The surface never reaches zero. [2, 3]

Maintenance

DefaultPoor

Server-side code, a live database and a plugin tree mean perpetual patching; a week of neglect can mean compromise. [1, 15]

TunedWeak

Managed services absorb the labor but never remove it — weekly update and staging cycles persist, and you pay for them indefinitely. [15, 16]

02

Jekyll

Static generator · Ruby · GitHub-native

The GitHub-native original — Ruby, born 2008, free to host on GitHub Pages. [9]

SEO

DefaultStrong

Pre-rendered pure HTML is what crawlers prefer; static output routinely scores 95+ Lighthouse where WordPress struggles to reach 90. [6, 7]

TunedStrong

Built-in sitemaps and RSS cover the SEO fundamentals; little is left to optimize. [8]

AEO

DefaultModerate

Clean HTML is a strong base, but Jekyll won't add schema or answer-first structure for you — that is manual work. [11, 13]

TunedStrong

Hand-authored semantic HTML, FAQ/Article schema and chunked, answer-first copy are fully under your control — exactly what answer engines reward. [11, 12]

Cost

DefaultStrong

Free hosting on GitHub Pages; the only spend is a domain (~$10–20/yr) and your time. [17]

TunedStrong

No runtime, no licenses — optimization is labor, not money.

Security

DefaultStrong

The served output is static files: no database, no server-side execution, almost nothing to exploit at runtime. [1]

TunedStrong

The only surface worth watching is build-time Ruby gems, never the live site.

Maintenance

DefaultModerate

The Ruby toolchain and gem dependencies are the friction point; builds slow as the site grows and GitHub Pages version-pinning can constrain you. [9, 10]

TunedModerate

Live-site security maintenance is near-zero; the recurring chore is the Ruby/gem build chain, which keeps this short of 'Strong.' [9]

03

Hugo

Static generator · single Go binary

A single Go binary with no dependency tree — it builds 1,000 pages in about two seconds. [6]

SEO

DefaultStrong

Pure static HTML with built-in sitemaps and RSS, and among the fastest builds in the category — which keeps Core Web Vitals effortless. [6, 8]

TunedStrong

Go templates give full control of head metadata and markup; a high ceiling, reached easily. [6]

AEO

DefaultModerate

Clean output, but structured data and answer-first patterns are yours to template in. [11, 13]

TunedStrong

Go templates make schema, semantic hierarchy and chunked answers straightforward to bake into every page. [11, 12]

Cost

DefaultStrong

Free static hosting; one binary, zero licenses. [17]

TunedStrong

Optimization costs time, not money.

Security

DefaultStrong

Static output plus a single self-contained binary — no runtime dependency tree to be compromised. [1]

TunedStrong

The smallest practical attack surface of the four; nothing executes server-side.

Maintenance

DefaultStrong

One binary means no dependency rot — no npm tree, no gem conflicts. Upgrades are a single file swap, and even 10,000 pages build in seconds. [6, 8]

TunedStrong

Hugo's signature advantage: the lowest long-run maintenance load in the set.

04

Astro

Static generator · islands · npm toolchain

Modern islands architecture that ships zero JavaScript by default — but reintroduces the npm toolchain. [7]

SEO

DefaultStrong

Islands hydrate only what is interactive, shipping zero JS for static content; ~95+ Lighthouse out of the box. [6, 7]

TunedStrong

The lightest payloads in this comparison; Core Web Vitals are a strength by design. [6]

AEO

DefaultStrong

Content collections and first-class structured-data support make schema and semantic markup native rather than bolt-on — arguably the best AEO ceiling here. [6, 12]

TunedStrong

Answer-first content, native schema and clean semantic HTML form the exact profile answer engines cite. [11, 14]

Cost

DefaultStrong

Free static hosting; no runtime cost. [17]

TunedStrong

Spend is zero; the real cost is build-toolchain attention (see Maintenance).

Security

DefaultStrong

Static output gives a near-zero runtime surface, like the other generators. [1]

TunedModerate

One honest caveat: the npm dependency tree is a build-time supply-chain surface the single-binary tools don't carry. The shipped site is safe; the build pipeline needs watching. [1]

Maintenance

DefaultModerate

npm means a real dependency tree — version churn and occasional breakage are the brittleness tax modern frameworks carry. [7]

TunedModerate

Excellent developer experience, but the toolchain needs periodic care; it won't match Hugo's single-binary durability.

03

The decision lens

Served architecture sets the floor. Everything else is configuration.

Three of these four are static generators, and that single fact collapses the two most expensive axes. With no live database and no server-side code, there is nothing to attack at runtime and nothing to pay a security team to defend — so cost and security fall to near-zero by default, before any tuning. WordPress is genuinely the most capable, but its economics run the other way: optimization adds cost and attack surface rather than removing them. You are buying capability you may not need with a recurring tax you cannot stop paying.

Among the static three, the split is durability versus modern developer experience — the owned-asset question. The mechanism is the dependency tree:

  • AHugo — a single Go binary, no dependency tree, the lowest long-run rot. This is the asset you can leave untouched for years and still rebuild on demand. If durability is the goal, it wins outright.
  • BJekyll — free and GitHub-native, served output as safe as any static site, but the Ruby/gem build chain is permanent friction that keeps maintenance off "Strong."
  • CAstro — the best AEO and authoring ceiling of the set, but it reintroduces the npm tree — exactly the brittleness a decades-horizon, hand-rolled philosophy is built to avoid. Worth it when the content model or interactivity earns the toolchain; otherwise it is rot you opted into.
  • DWordPress — reach for it only when non-technical editing, a plugin ecosystem, or commerce is the actual requirement. Then pay the tax knowingly.
